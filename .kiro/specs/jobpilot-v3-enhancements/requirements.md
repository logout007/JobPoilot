# Requirements Document

## Introduction

JobPilot V3 enhances the existing V2 "Find & Recommend" platform with four pillars:

1. **CV Auto-Tailoring** — Gemini generates a role-specific CV snippet and cover letter opening for every evaluated job, stored in S3 and surfaced in the dashboard.
2. **Application Tracking** — Users add timestamped notes to any job, set follow-up reminder dates, and receive SES reminder emails when deadlines approach.
3. **Analytics Dashboard** — A dedicated analytics view shows grade trends over time, per-platform performance, and funnel metrics (found → reviewed → applied).
4. **Scraper & Evaluator Hardening** — Retry logic with exponential back-off for all scrapers, improved Gemini prompt quality, and a professional monorepo file structure.

The system continues to run on AWS free-tier services (Lambda, DynamoDB, SES, S3, SSM, EventBridge, API Gateway) with a target monthly cost of $0.00.

---

## Glossary

- **CV_Snippet**: A 150–200 word tailored summary paragraph generated by Gemini that positions the candidate specifically for a given job role.
- **Cover_Opening**: A 2–3 sentence cover letter opening generated by Gemini, referencing the company and role by name.
- **Tailoring_Package**: The combined CV_Snippet + Cover_Opening stored as a single markdown file in S3.
- **Note**: A user-authored text entry attached to a Job_Record with a creation timestamp.
- **Reminder**: A scheduled follow-up date attached to a Job_Record; triggers an SES email on that date.
- **Analytics**: Aggregate metrics computed from Job_Records covering grade distribution, platform performance, and application funnel.
- **Funnel**: The progression of a job through statuses: New → Reviewed → Applied.
- **Retry**: Re-attempting a failed scraper page load or Gemini API call using exponential back-off.
- **Monorepo**: A single repository with source code organized into `src/` subdirectories per Lambda, shared utilities in `src/shared/`, infrastructure in `infra/`, tests in `tests/`, and static assets in `public/`.

## Requirements

### Requirement 1: CV Auto-Tailoring — Snippet Generation

**User Story:** As a job seeker, I want a tailored CV snippet generated for each job, so that I can quickly customize my application without starting from scratch.

#### Acceptance Criteria

1. WHEN the Job_Evaluator processes a job, THE Job_Evaluator SHALL call Gemini to generate a CV_Snippet of 150–200 words that repositions the candidate's experience to match the job's key requirements.
2. THE CV_Snippet SHALL reference at least 2 specific skills or projects from the candidate's CV that are directly relevant to the job description.
3. THE CV_Snippet SHALL be written in first person, professional tone, and be ready to paste directly into a CV summary section.
4. THE Job_Evaluator SHALL generate a Cover_Opening of 2–3 sentences that addresses the company by name and references the specific role title.
5. THE Job_Evaluator SHALL combine the CV_Snippet and Cover_Opening into a single Tailoring_Package markdown file.
6. THE Job_Evaluator SHALL upload the Tailoring_Package to S3 with path `/tailoring/{YYYY-MM-DD}/{jobId}.md`.
7. THE Job_Evaluator SHALL store the S3 URL of the Tailoring_Package in the Job_Record as `tailoringUrl`.
8. IF Gemini fails to generate the tailoring content, THE Job_Evaluator SHALL log the error and store an empty `tailoringUrl` without failing the job evaluation.

### Requirement 2: CV Auto-Tailoring — Dashboard Integration

**User Story:** As a job seeker, I want to view and copy my tailored CV snippet directly from the dashboard, so that I can apply quickly without switching tools.

#### Acceptance Criteria

1. WHEN a job card has a `tailoringUrl`, THE Dashboard SHALL display a "View Tailoring" button alongside the existing action buttons.
2. WHEN the user clicks "View Tailoring", THE Dashboard SHALL fetch the Tailoring_Package markdown from S3 and render it in a modal with proper formatting.
3. THE modal SHALL include a "Copy CV Snippet" button that copies only the CV_Snippet section to the clipboard.
4. THE modal SHALL include a "Copy Cover Opening" button that copies only the Cover_Opening section to the clipboard.
5. WHEN content is copied, THE Dashboard SHALL show a "Copied!" toast notification for 2 seconds.
6. THE "View Tailoring" button SHALL be visually distinct (purple/violet color) from other action buttons.

---

### Requirement 3: Application Tracking — Notes

**User Story:** As a job seeker, I want to add notes to any job, so that I can record interview feedback, recruiter contacts, and personal observations.

#### Acceptance Criteria

1. THE Dashboard SHALL display a "Notes" section on each job card that is expandable/collapsible.
2. WHEN the Notes section is expanded, THE Dashboard SHALL show all existing notes for that job in chronological order, each with its creation timestamp.
3. THE Dashboard SHALL provide a text input and "Add Note" button within the Notes section.
4. WHEN the user submits a note, THE Dashboard SHALL call the API to persist the note and update the UI immediately.
5. EACH note SHALL be stored as an object with fields: `text` (string), `createdAt` (ISO 8601 timestamp).
6. THE System SHALL store notes as a list attribute `notes` on the Job_Record in DynamoDB.
7. THE API SHALL support a `POST /jobs/{jobId}/notes` endpoint that appends a note to the Job_Record.
8. THE API SHALL return the updated notes list in the response.
9. IF the note text is empty or whitespace-only, THE Dashboard SHALL not submit the note and SHALL show a validation message.

---

### Requirement 4: Application Tracking — Reminders

**User Story:** As a job seeker, I want to set follow-up reminders on jobs, so that I never miss a deadline or forget to follow up after applying.

#### Acceptance Criteria

1. THE Dashboard SHALL display a "Set Reminder" button on each job card.
2. WHEN the user clicks "Set Reminder", THE Dashboard SHALL show a date picker input for selecting the reminder date.
3. WHEN the user confirms a reminder date, THE Dashboard SHALL call the API to persist the reminder and update the UI.
4. THE System SHALL store the reminder date as a `reminderDate` attribute (ISO 8601 date string) on the Job_Record.
5. THE API SHALL support a `PUT /jobs/{jobId}/reminder` endpoint that sets or updates the `reminderDate`.
6. THE Job_Evaluator Lambda SHALL query DynamoDB daily for Job_Records where `reminderDate` equals today's date.
7. FOR each job with a due reminder, THE System SHALL send an SES email to the notification address containing the job title, company, platform, current status, and a link to the dashboard.
8. AFTER sending the reminder email, THE System SHALL clear the `reminderDate` field on the Job_Record to prevent duplicate sends.
9. IF the SES send fails for a reminder, THE System SHALL log the error and retain the `reminderDate` for retry on the next run.
10. THE Dashboard SHALL display an active reminder date badge on job cards that have a pending reminder.

### Requirement 5: Analytics — Grade Trends Over Time

**User Story:** As a job seeker, I want to see how job quality trends over time, so that I can understand whether the market is improving or if my profile needs updating.

#### Acceptance Criteria

1. THE Analytics view SHALL display a line chart showing the daily count of A, B, C, D, and F grade jobs over the last 30 days.
2. THE Analytics view SHALL display the average total score per day over the last 30 days.
3. THE data for the grade trend chart SHALL be computed from Job_Records grouped by the `foundAt` date field.
4. THE Analytics view SHALL display the overall grade distribution as a bar chart (count per grade across all time).
5. THE API SHALL provide a `GET /analytics/grades` endpoint that returns grade counts grouped by date for the last 30 days.

---

### Requirement 6: Analytics — Platform Performance

**User Story:** As a job seeker, I want to see which platforms deliver the best job quality, so that I can focus my attention on the most productive sources.

#### Acceptance Criteria

1. THE Analytics view SHALL display a table showing per-platform metrics: total jobs found, average grade score, count of A-grade jobs, and application conversion rate (Applied / Total).
2. THE Analytics view SHALL highlight the top-performing platform (highest average score) with a visual indicator.
3. THE Analytics view SHALL show the scraper success rate per platform (successful scrapes / total attempts) over the last 7 days.
4. THE API SHALL provide a `GET /analytics/platforms` endpoint that returns per-platform aggregated metrics.

---

### Requirement 7: Analytics — Application Funnel

**User Story:** As a job seeker, I want to see my application funnel metrics, so that I can understand my conversion rates and identify bottlenecks.

#### Acceptance Criteria

1. THE Analytics view SHALL display a funnel visualization showing counts at each stage: Found → Reviewed → Applied.
2. THE Analytics view SHALL display the conversion rate between each funnel stage as a percentage.
3. THE Analytics view SHALL display the total count of Rejected and Archived jobs separately.
4. THE Analytics view SHALL display the average time from "New" to "Applied" in days.
5. THE API SHALL provide a `GET /analytics/funnel` endpoint that returns funnel stage counts and conversion rates.

---

### Requirement 8: Analytics — Dashboard Navigation

**User Story:** As a job seeker, I want seamless navigation between the jobs view and analytics view, so that I can switch contexts without losing my place.

#### Acceptance Criteria

1. THE Dashboard SHALL have a top navigation bar with two tabs: "Jobs" and "Analytics".
2. WHEN the user clicks "Analytics", THE Dashboard SHALL switch to the analytics view without a full page reload.
3. WHEN the user clicks "Jobs", THE Dashboard SHALL return to the jobs list view, preserving any active filters.
4. THE active tab SHALL be visually highlighted.
5. THE Dashboard SHALL remember the last active tab across page refreshes using localStorage.

---

### Requirement 9: Scraper Hardening — Retry Logic

**User Story:** As a developer, I want scrapers to retry on transient failures, so that temporary network issues or bot-detection challenges don't cause entire platforms to be skipped.

#### Acceptance Criteria

1. EACH platform scraper SHALL retry up to 3 times on page load timeout or navigation error before returning an empty array.
2. THE retry delay SHALL use exponential back-off: 2s, 4s, 8s between attempts.
3. WHEN a scraper succeeds on a retry attempt, THE System SHALL log the attempt number and platform name.
4. WHEN all retry attempts are exhausted, THE System SHALL log the final error with attempt count and return an empty array.
5. THE retry logic SHALL be implemented in a shared `withRetry(fn, maxAttempts, baseDelayMs)` utility function.
6. THE `withRetry` function SHALL be usable for both scraper page loads and Gemini API calls.

---

### Requirement 10: Scraper Hardening — Selector Resilience

**User Story:** As a developer, I want scrapers to use resilient selector strategies, so that minor DOM changes on job platforms don't break extraction.

#### Acceptance Criteria

1. EACH scraper SHALL use a prioritized selector list: try the primary selector first, fall back to secondary selectors if the primary returns no results.
2. WHEN a fallback selector is used, THE System SHALL log a warning with the platform name and which selector was used.
3. EACH scraper SHALL validate that extracted job objects have at minimum a non-empty `title` and `url` before including them in results.
4. Jobs failing validation SHALL be logged with the platform name and the missing fields, then discarded.
5. THE selector lists for each platform SHALL be defined as named constants at the top of the scraper module, not inline in the extraction logic.

---

### Requirement 11: Gemini Prompt Quality Improvements

**User Story:** As a job seeker, I want more accurate and consistent AI evaluations, so that the grades and reports are reliable and actionable.

#### Acceptance Criteria

1. THE evaluation prompt SHALL include explicit scoring rubrics for each dimension (e.g., "Score 5 if salary exceeds minimum by 20%+, score 4 if within 10%, score 3 if at minimum...").
2. THE evaluation prompt SHALL include 2 few-shot examples showing a high-scoring job and a low-scoring job with expected JSON output.
3. THE evaluation prompt SHALL instruct Gemini to output ONLY valid JSON with no markdown fences, preamble, or explanation.
4. THE Job_Evaluator SHALL validate the parsed JSON against a schema (all 10 dimension keys present, all values 0–5 numeric) before accepting it.
5. IF schema validation fails, THE Job_Evaluator SHALL retry the Gemini call once with an explicit correction prompt before falling back to defaults.
6. THE tailoring prompt SHALL include the full job description (not truncated) and the candidate's top 5 most relevant skills identified from the evaluation.

---

### Requirement 12: Professional Monorepo File Structure

**User Story:** As a developer, I want a clean, professional project structure, so that the codebase is maintainable, navigable, and follows industry conventions.

#### Acceptance Criteria

1. THE project SHALL organize source code into `src/` with subdirectories: `src/scanner/`, `src/evaluator/`, `src/api/`, `src/shared/`.
2. THE `src/shared/` directory SHALL contain utilities used by multiple Lambdas: `dynamo.js`, `s3.js`, `ssm.js`, `ses.js`, `retry.js`, `selectors.js`.
3. THE `src/scanner/` directory SHALL contain: `index.js` (handler entry point), `scrapers/` (one file per platform), `screenshot.js`, `extractor.js`.
4. THE `src/evaluator/` directory SHALL contain: `index.js` (handler entry point), `evaluator.js`, `grader.js`, `reporter.js`, `tailoring.js`, `reminders.js`.
5. THE `src/api/` directory SHALL contain: `index.js` (handler entry point), `routes/jobs.js`, `routes/analytics.js`.
6. THE `infra/` directory SHALL contain all Terraform files: `main.tf`, `variables.tf`, `outputs.tf`, `terraform.tfvars`.
7. THE `tests/` directory SHALL mirror the `src/` structure: `tests/scanner/`, `tests/evaluator/`, `tests/api/`, `tests/shared/`.
8. THE `public/` directory SHALL contain static assets: `dashboard.html` (renamed from `dashboard-v2.html`).
9. ALL legacy root-level `.js` handler files (`handler.js`, `scanner-handler.js`, `evaluator-handler.js`, `api-handler.js`, `dashboard-api.js`) SHALL be migrated into the new structure and the originals deleted.
10. ALL legacy root-level markdown documentation files SHALL be consolidated into a single `docs/` directory.
11. THE `package.json` build scripts SHALL be updated to reference the new file paths.
12. THE `main.tf` Lambda `filename` and `handler` references SHALL be updated to match the new structure.
